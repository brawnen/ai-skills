## 适用场景（与 `~/.claude/CLAUDE.md` 触发条件对齐）

满足任一条件即触发：

- 跨文件修改 ≥ 3 个
- 涉及 schema / 对外接口 / 配置文件 / 不可逆操作
- 用户明确要求"先列 TODO"或"按 TODO 执行"

## 不适用场景

- 单文件局部修改且无副作用
- 纯问答、纯探索、纯只读分析
- 一行 typo 或注释级修复

## 核心规则

1. 执行前先确认目标、边界、影响范围和验证方式。
2. 先形成 TODO，再开始改文件或跑有副作用的命令。
3. TODO 至少包含：状态、步骤、涉及范围、验证方式、备注。
4. 同一时间只保留一个核心步骤为"执行中"；并行子任务允许，但必须显式标注"并行"。
5. 每完成一步立即更新 TODO 状态，不要到最后一次性补状态。
6. 遇到阻塞必须标记"阻塞"，并说明阻塞原因、影响范围、可选下一步。
7. 未验证的步骤不能标记为"已完成"，只能标记为"待验证"或说明验证受限。
8. TODO 中途调整时，保留原条目并标注"调整原因"，不要直接删除已开始过的步骤。

## 状态值

统一使用以下状态，避免随意扩展：

- 待执行
- 执行中
- 已完成
- 待验证
- 阻塞
- 跳过

## TODO 落盘位置

- **短任务**：使用宿主原生任务工具或对话内 TODO，不必落盘。
- **中大型任务**：落盘 Markdown TODO 文档。
- **落盘路径优先级**：
  1. 项目 `AGENTS.md` / `CLAUDE.md` 指定的路径
  2. 项目已有的 TODO / docs 约定目录
  3. 兜底 `.tasks/` 或与设计文档同目录的 `*-todolist.md`

不要为简单任务强行创建文档；不要把 TODO 分散到多个位置导致状态不一致。

## TODO 模板（落盘时使用）

```markdown
# 任务 TODO

## 目标
说明本次任务要解决什么问题。

## 边界
- 本次处理：
- 本次不处理：
- 关键风险：

## TODO List

| 状态 | 步骤 | 涉及范围 | 验证方式 | 备注 |
| --- | --- | --- | --- | --- |
| 待执行 | 1. 梳理现有入口和影响范围 | 代码 / 文档路径 | 代码阅读 / 命令检查 |  |
| 待执行 | 2. 实现最小改动 | 目标模块 | 编译 / 单测 / 脚本 |  |
| 待执行 | 3. 验证与回填 | 测试 / 文档 | 验证命令 |  |
```

## 执行流程

1. 先列 TODO，并把第一步标记为"执行中"。
2. 执行当前步骤，只做该步骤必要范围内的改动。
3. 完成后立即更新该步骤状态。
4. 验证未跑或无法跑，标记为"待验证"并写明原因。
5. 进入下一步前，把下一步标记为"执行中"。
6. 中途新增 / 合并 / 跳过任务时，保留原条目并标注"调整原因"。

## 任务收尾

任务结束时**不要在本 skill 内自写汇报**，调用 `task-report` skill 完成统一格式的最终输出。`task-report` 是所有汇报的唯一出口。
