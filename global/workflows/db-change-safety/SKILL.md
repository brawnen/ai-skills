---
name: db-change-safety
description: 适用于数据库相关改动，包括 DDL、索引、迁移、回填、数据修复及上生产的 SQL 调整；目标是优先识别风险再决定方案。纯只读 SELECT / EXPLAIN / 方案讨论阶段不触发。
---

## 适用场景

- 修改表结构、字段、索引、约束（schema / index change）
- 编写或修改 migration
- 写数据修复脚本、回填脚本、批处理 SQL（data migration / data fix）
- 优化将上生产的 SQL 或分析潜在数据库风险
- 任何写入生产数据库的不可逆操作

## 不适用场景

- 纯只读 SELECT / EXPLAIN / 慢查询分析（无写入）
- 方案讨论阶段，未进入落地
- 本地开发库的临时 DDL 实验
- ORM 标准查询（无手写 SQL、无 N+1 风险）

## 与全局规则的关系

- 命中本 skill 通常同时命中 `~/.claude/CLAUDE.md` 的"不可逆操作"触发条件
- 因此本 skill 通常与 `task-todolist-execution` 同时出现，最终由 `task-report` 收尾

## 与其它 skill 协作

- 本 skill 的"主要风险"清单 → 直接填到 `task-todolist-execution` 的"边界 / 关键风险"段
- 本 skill 的"回滚 / 兜底方案"与"验证建议" → 在 `task-report` 的"遗留风险与后续"段单独列出
- 本 skill 不替代 `task-report`，最终汇报仍走 `task-report` 唯一出口

## 环境分级

执行前必须先识别目标环境：

| 环境 | 检查强度 | 备注 |
| --- | --- | --- |
| 本地开发库 | 可降级，仅保留改动分类和回滚思路 | 不强制预检清单 |
| Staging | 走完整流程，允许试错 | 数据修复必须可重跑 |
| 生产 | 完整流程 + 显式回滚 SQL + 灰度 / 分批 | 任何环节不齐都不上线 |

## 改动分类与必检风险

第一步必须先识别本次属于哪一类，并对照下表逐项检查：

| 改动类型 | 必检风险 |
| --- | --- |
| schema change | 锁表时长 / 在线 DDL 可行性 / 旧新代码兼容窗口 / 默认值与 NULL 处理 |
| index change | 在线创建可行性 / 索引膨胀 / 现有 SQL 执行计划是否退化 |
| query change | 是否走索引 / 是否全表扫描 / 是否触发文件排序 / 缓存影响 |
| data migration | 幂等性 / 批次大小 / 失败恢复 / 回滚 SQL / 与业务写入的并发冲突 |
| data fix | 影响行数预估 / 备份 / 灰度 / KILL 预案 / 是否有审计需求 |

## 预检清单（生产环境强制）

执行前必须逐项确认：

- [ ] 已确认目标数据库及版本（MySQL / PostgreSQL / SQLite / 其它）
- [ ] `EXPLAIN` 确认走索引、无意外全表扫描
- [ ] `SELECT COUNT(*)` 预估影响行数
- [ ] 测试库或 staging 完成 dry-run
- [ ] 准备好可执行的回滚 SQL（不是"理论可回滚"）
- [ ] 大规模数据变更已分批 + 限速
- [ ] 准备好长事务 KILL 预案
- [ ] 与业务写入并发的影响已评估

## 核心规则

1. 先判断改动类型，再对照"必检风险"清单逐项分析。
2. 先给出风险判断和影响范围，再给出方案。
3. 数据修改必须考虑幂等性、回滚性、执行批次、失败恢复。
4. schema 变更必须考虑兼容窗口和旧 / 新代码共存问题。
5. 输出中必须单独列出风险与验证建议，不能与方案混写。
6. 不同数据库方言差异（DDL 事务性、在线 ALTER 能力、SQLite ALTER 限制等）必须在分析中显式提及。

## 禁止事项

- 不要默认数据库变更是低风险操作
- 不要把 schema 变更和数据回填混成一个模糊动作
- 不要在未说明风险时直接给"可上线"结论
- 不要忽略锁表、长事务、全表扫描或索引失效风险；未使用索引的 SQL 必须提前告知
- 不要把"语法正确"等同于"线上安全"
- 未确认幂等性前，不要声称脚本可重复执行
- 未确认回滚思路前，不要默认可轻易回退
- **业务代码中禁止裸 DELETE 和无 WHERE 的 UPDATE**；如确需硬删除或全表更新，必须以独立的迁移脚本 / 运维脚本形式提交，并显式标注「需人工确认」「需备份」

## 输出结构

严格按以下结构输出（无内容写"无"）：

### 改动类型
- [schema change / index change / query change / data migration / data fix；多选请分别列出]

### 目标环境与数据库
- [本地 / staging / 生产；MySQL 8.0 / PostgreSQL 15 / SQLite 等及版本]

### 影响范围
- [涉及的表、查询、读写路径、上下游业务、缓存]

### 主要风险
- [对照"必检风险"清单的实际命中项，给出具体描述]

### 回滚 / 兜底方案
- [可执行的回滚 SQL 或回滚步骤；如无法回滚必须显式说明]

### 验证建议
- [EXPLAIN、count、dry-run、灰度、分批、KILL 预案等的具体动作]

### 确定性 / 不确定性
- [哪些结论是基于事实的，哪些是基于假设的；假设需要后续确认]
