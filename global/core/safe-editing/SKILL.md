## Use when
- 用户要求修改已有代码，特别是维护多年的历史系统
- 任务是修 bug、补丁、局部优化、兼容修正或配置修正
- 需要在已有仓库中做小到中等规模改动，且测试覆盖较弱或验证成本高
- 仓库较复杂，扩大改动范围会显著增加风险

## Do
1. 先定位与需求直接相关的文件、模块、函数或配置项
2. 优先采用局部修复和最小闭环修改，而不是扩展成架构重构或全盘整理
3. 只修改完成当前任务所必需的代码路径
4. 如发现潜在更大问题，先记录为 follow-up，而不是直接展开或顺手修改
5. 在输出中说明为什么只改这些文件，以及如何控制了风险

### 对于历史遗留系统 (Legacy Code) 额外要求
- 默认优先保持现有行为和兼容性，修改前先识别高风险区域（如老配置、公共入口、老事务）
- 尽量复用现有模式，而不是引入新框架、新范式或新依赖
- 不要假设老系统一定支持新工具链或新规范
- 不要改变外部接口行为，除非用户明确要求

## Rules
- 不要为了“更优雅”或“现代化”而扩大改动范围
- 不要混入无关格式化、命名调整、目录整理或依赖升级
- 不要轻易替换已有基础库、框架写法或项目模式
- 不要在一次任务中同时处理多个历史问题，也不要同时做功能修改和大重构
- 不要因为顺手就改 shared config、root config 或全局脚本
- 如果必须扩大范围或引入新依赖，必须先说明原因和影响面并获得同意

## Validation
- 检查改动文件是否都与当前任务直接相关，是否保持在局部范围内
- 检查是否混入了无关改动（包括新的依赖、全局配置变更）
- 检查是否改变了现有接口、配置或关键兼容性行为

## Output
- Changed files & Scope control choices
- Why these files (and Legacy risk areas touched if applicable)
- Compatibility considerations
- Follow-up issues noticed
