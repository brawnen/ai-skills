# Production Incident Hotfix

## Use when

- 用户描述线上被攻击、短信/验证码/登录等接口异常、部署任务卡住、第三方接口失败但页面假成功。
- 用户要求“先分析原因”“先来个临时方案”“现在怎么止血”“发布步骤”。
- 需要在有限时间内控制损失，同时不把临时方案包装成长期方案。
- 性能回退、非确定性故障、线上偶发错误需要先建立可观测反馈回路。

## Coordinate with existing skills

- 先用 `systematic-debugging` 的证据优先原则定位根因。
- 修改老系统时同时遵守 `safe-editing`。
- 涉及 DB 写入时先用 `db-change-safety`。
- 需要补回归测试时配合 `test-first-development`。
- 收尾用 `task-report` 明确已验证/未验证。

## Workflow

1. 保存原始症状和报错，不先改代码。
2. 定义事件边界：影响接口、用户路径、时间窗口、环境、是否已上线、是否仍在扩大。
3. 建立最小反馈回路，优先级从强到弱：
   - 真实日志、traceId、监控指标、告警时间线。
   - 可重复请求：curl/HTTP 脚本、接口参数、HAR、消息 payload。
   - 自动化复现：单测、集成测试、CLI fixture、Playwright、trace replay、最小 harness。
   - 人工复现：明确步骤、输入、观察点和结果记录。
   如果无法建立反馈回路，先说明尝试过什么，并向用户要日志、HAR、录屏、环境访问或临时观测点授权。
4. 优先看真实日志或可观测信号，再反查代码入口；没有日志时说明证据缺口。
5. 枚举所有候选入口，不只看第一个命中的 controller 或页面。
6. 如果项目存在 `.codegraph/` 或用户要求使用 CodeGraph，用它加速代码侧归因：
   - `codegraph query <入口/符号>` 定位 controller、service、vendor client、测试。
   - `codegraph callers <symbol>` / `codegraph callees <symbol>` 拼第一版传播链路。
   - `codegraph context "<事件描述>"` 生成初始上下文包。
   - 日志序列、真实请求参数、页面调用顺序和源码复核优先级高于静态图谱。
7. 生成 3-5 个可证伪假设，并按概率/风险排序。每个假设必须写清预测：
   - 如果 X 是原因，那么观察/修改/压测 Y 后，症状会如何变化。
   - 不能写预测的假设先丢弃或重新表述。
   - 用户在线时先给出排序，允许用户用业务/上线信息重排。
8. 只为验证假设加探针：
   - 一次只改一个变量。
   - 优先 debugger/REPL/已有监控；其次才是边界日志。
   - 临时 debug 日志必须带唯一前缀，例如 `[DEBUG-a4f2]`，方便最后清理。
   - 性能问题先建立基线指标或 profile，不用日志猜性能瓶颈。
9. 用特征链路收敛来源，例如“先请求图形验证码再请求短信验证码”“页面调用顺序”“traceId”“供应商返回码传播”。
10. 判断根因层级：
    - 前端入口暴露
    - 后端校验缺失
    - 编排层过滤/传播错误
    - 第三方 vendor 失败未抛出
    - 部署脚本会话/IO 未释放
    - 环境、依赖、容量或性能问题
11. 给出临时止血方案，并单列绕过面和误伤面。
12. 给出长期方案，必须比临时特征更强，例如服务端可验证票据、行为验证码、幂等/补偿、供应商侧风控、流程重构、架构测试缝补齐。
13. 修复后复跑原始反馈回路；如果存在正确测试缝，补回归测试。
14. 收尾清理所有临时探针、debug 日志和 throwaway harness；如果无法清理，明确遗留位置和原因。
15. 如用户要求发布，转入 `project-release-scope-check`。

## Guardrails

- 不用接口名相似度直接判断攻击来源；优先日志序列和调用链。
- 不把 Origin、Referer、UA、改路径、单 IP 阈值当强防护。
- 不把 warning-free 当成功标准；验收标准应贴近事件目标。
- 不能因为页面提示成功就认定 vendor 授权成功。
- 不把 CodeGraph 静态调用链当作根因证据；根因仍需日志、源码和复现/验证支撑。
- 没有反馈回路时，不直接跳到单一根因结论；最多输出候选和证据缺口。
- 不“log everything and grep”；探针必须对应具体假设。
- 临时 debug 日志、探针、脚本必须在收尾阶段清理或登记遗留。
- 临时止血不是长期方案；必须说明绕过面、误伤面和回滚方式。
