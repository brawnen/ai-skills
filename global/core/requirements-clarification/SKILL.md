# Requirements Clarification

## Goal

把模糊需求、方案或 P0 问题变成可执行、可验证、可交接的 spec。先建立共同理解，再决定是否进入实现、排障、发布核对或文档沉淀。

这个 skill 结合三类流程：

- `brainstorming` 的项目上下文探索、2-3 个方案对比和设计确认。
- `grill-me` 的逐题追问：一次只问一个关键问题，并给推荐答案。
- `grill-with-docs` 的术语澄清、代码交叉验证和必要决策沉淀。

## Default Mode

- 默认先澄清，再落盘 spec。
- 用户明确说“只分析 / 先不要改 / 先聊 / 不落盘”时，不写文件，只输出 spec 草案。
- 不默认 `git commit`，除非用户明确要求。
- 不默认进入实现；实现前应转入 `task-todolist-execution`、`safe-editing`、`test-first-development` 或相关专项 skill。

## Workflow

1. 确认目标和模式：本次是需求澄清、方案评审、P0 边界确认、接口设计、流程设计，还是重构前置讨论。
2. 探索可验证上下文：
   - 读已有 README、AGENTS、docs、接口定义、测试、配置、相关源码。
   - 如果项目有 `.codegraph/` 或用户要求，配合 `codegraph-assisted-navigation` 找候选入口和调用链。
   - 能从代码/配置确认的问题先查证，不把可验证事实直接丢给用户。
3. 建立证据分层：
   - 已确认：来自代码、配置、文档或用户明确回答。
   - 推断：来自调用关系、命名、相似实现或上下文。
   - 待确认：必须由用户、产品、运维、DBA 或外部系统确认。
4. 逐题澄清：
   - 一次只问一个关键问题。
   - 每个问题给推荐答案和理由。
   - 优先问会阻塞后续设计的问题，不问装饰性问题。
   - 用户回答后再进入下一个依赖分支。
5. 方案对比：当存在多个可行路径时，给 2-3 个方案，说明成本、风险、验证方式和推荐方案。
6. 形成 spec 草案，先让用户确认关键章节。
7. 用户确认后落盘 spec：
   - 优先使用项目已有 spec/docs 约定。
   - 若无约定，写入 `docs/specs/YYYY-MM-DD-<topic>-spec.md`。
   - 如果是 P0/事故方案，也可以写入 `docs/runbooks/`，但需说明原因。
8. 落盘后做自检：占位符、矛盾、范围蔓延、未确认项、验证口径是否清楚。
9. 输出下一步：实现、测试、发布核对、排障、或继续澄清。

## Spec Template

```markdown
# <标题>

## 背景
- 问题/目标：
- 当前上下文：
- 非目标：

## 已确认事实
- ...

## 待确认问题
- ...

## 方案
### 推荐方案
- 做法：
- 原因：
- 影响范围：
- 风险：

### 备选方案
- ...

## 接口 / 数据 / 流程影响
- ...

## 验证方式
- 自动化验证：
- 人工验证：
- 未覆盖风险：

## 执行边界
- 本次处理：
- 本次不处理：
- 进入实现前置条件：
```

## Guardrails

- 不把推断写成已确认事实。
- 不一次性抛出多个问题轰炸用户。
- 不为了完整性强行扩大范围；发现大问题先作为非目标或后续项。
- 不在用户只读意图下写文件。
- 不把 spec 当最终实现计划；实现计划应另行拆 TODO。
- 不创建 ADR，除非决策难回退、没有上下文会困惑、并且确实有取舍。

## Coordination

- 需要正式代码证据文档：配合 `code-evidence-docs`。
- 后续进入实现：配合 `task-todolist-execution` 和 `safe-editing`。
- 后续涉及测试优先：配合 `test-first-development`。
- 生产事故或攻击：配合 `production-incident-hotfix`，本 skill 只负责把问题边界和方案确认清楚。
- 发布范围核对：配合 `project-release-scope-check`。
