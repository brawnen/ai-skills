## Use when
- 任务结束准备汇报，或用户要求总结改动
- 做了任何代码、配置、脚本、SQL 等修改后，准备声明“已完成”
- 需要给后续接手的人快速说明上下文及变更影响面

## Do (强制验证前置)
1. **绝对前置条件：必须执行验证。** 优先运行相关的自动化测试、构建或检查命令。如果没有自动化测试，必须提供最小人工验证步骤。
2. 用固定的 Markdown 结构总结结果，优先写事实，不写空话。
3. 如果环境限制导致无法验证，必须明确说明受阻原因。
4. 区分“已验证部分”和“未验证部分”。
5. 输出中只写实际执行过的验证动作，不写假设性验证。

## Rules
- 未经实际验证前，绝对不要声称“已修复”、“已完成”或“可以提交”
- 不要把“代码看起来没问题”等同于“验证通过”
- 不要输出泛泛而谈的总结，验证失败时必须明确失败点
- 不要隐藏剩余风险，不要把推测写成事实
- 内容要简洁、可扫描、可交接

## Output Structure
请严格按照以下结构输出汇报（如果没有阻塞或风险可写 None）：

### What I changed
- [文件列表/核心逻辑摘要]

### Why
- [原因简述，非技术流水账]

### Validation
- **Action**: [执行了什么验证命令 / 人工测试步骤]
- **Result**: [验证结果事实]
- **Unverified**: [哪些边界或路径未能验证]

### Risks / Follow-ups / Blocking Issues
- [潜在风险、遗留问题或阻塞验证的原因]
