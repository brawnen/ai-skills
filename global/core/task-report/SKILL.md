## 适用场景

- 任务结束准备汇报，或用户要求总结改动
- 做了任何代码、配置、脚本、SQL 等修改后，准备声明"已完成"
- 需要给后续接手的人快速说明上下文及变更影响面
- 作为 `task-todolist-execution` 的统一汇报出口

## 不适用场景

- 纯问答、纯只读分析、纯探索阶段
- 单行 typo 或注释级修复
- 用户明确说"先不要总结"

## 与全局规则的关系

- 对齐 `~/.claude/CLAUDE.md` 的"验证纪律"段
- 所有声明"已完成 / 已修复 / 可提交"必须经过本 skill 的输出结构

## 强制规则

1. **绝对前置条件：必须执行验证。** 优先运行相关的自动化测试、构建或检查命令；没有自动化测试时，必须提供最小人工验证步骤。
2. 用固定的 Markdown 结构总结结果，优先写事实，不写空话。
3. 区分"已验证部分"和"未验证部分"。
4. 只写实际执行过的验证动作，不写假设性验证。
5. 如果存在阻塞，结论不能写"已完成"，必须先把阻塞写清楚。
6. 区分"阻塞"和"遗留风险"：阻塞影响当前是否交付，遗留风险影响后续工作。

## 禁止事项

- 未经实际验证前，绝对不要声称"已修复"、"已完成"或"可以提交"
- 不要把"代码看起来没问题"等同于"验证通过"
- 不要把推测写成事实
- 不要把"阻塞"和"遗留风险"混为一谈
- 不要输出泛泛而谈的总结，验证失败时必须明确失败点

## 输出结构

严格按以下结构输出（无内容写"无"）：

### 改动内容
- [文件列表 / 核心逻辑摘要]

### 改动原因
- [原因简述，非技术流水账]

### 验证
- **执行动作**：[验证命令 / 人工测试步骤]
- **结果**：[验证结果事实，保留原始报错]
- **未验证**：[哪些边界或路径未能验证]

### 阻塞
- [阻塞验证或交付的原因、影响范围、可选下一步；无则写"无"]

### 遗留风险与后续
- [验证通过但未来可能出问题的点，或下一步可做的优化；无则写"无"]
